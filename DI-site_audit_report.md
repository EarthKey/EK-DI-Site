## 2. 入力検証・出力制御（2.1〜2.2）

| 項目 | 評価 | リスク | 備考 |
|------|------|--------|------|
| 検索入力のXSS | **適合** | Low | `searchInput`の値はDOM表示に使われず、`.includes(keyword)` でフィルタリングのみ。HTMLへの書き出しなし |
| innerHTML使用箇所 | **要注意** | Medium | `renderCategories()` 内で `article.innerHTML` に `${item.url}`, `${item.name}`, `${item.use}` を直接展開している。**現在はデータがファイル内ハードコードのため問題なし**。将来、外部APIやユーザー入力からデータを取る設計に変わった場合は即 XSS 脆弱性になる。 |

**改善提案：**
将来的なデータ外部化に備え、以下のようなエスケープ関数を用意しておくと安全。
```js
function esc(str) {
  return String(str).replace(/[&<>"']/g, c =>
    ({ '&':'&amp;','<':'&lt;','>':'&gt;','"':'&quot;',"'":'&#39;' }[c])
  );
}
```

## 3. HTTPセキュリティヘッダー（3.1〜3.2）

| 項目 | 評価 | リスク | 備考 |
|------|------|--------|------|
| HSTS | 該当なし | — | ホスティング側（GitHub Pages）で設定済み |
| CSP（Content-Security-Policy） | **不適合** | Medium | `<meta http-equiv="Content-Security-Policy">` が未設定。外部スクリプト読み込みはないが、明示的に制限しておくと堅牢 |
| X-Frame-Options / フレーム制御 | 該当なし | — | 静的ファイル。ホスティング側で対応 |

**改善提案（優先度 Medium）：**
```html

  content="default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; frame-ancestors 'none';">
```
ただし `unsafe-inline` を使っているため、将来的にはインラインスクリプト・スタイルを外部ファイル化してnonceベースCSPに移行するのが理想。

## 17. 設計品質の追加評価

### 17.1 情報設計の強み

このサイトの最大の強みは、一般的な「カテゴリ列挙型リンク集」ではなく、**現場の困りごとから逆引きできる導線設計**になっている点である。

特に以下は評価できる。

1. **状況起点のトップ導線**
   - 「まず原典を押さえたい」「授乳・ドーピングで困る」など、利用者の認知に近い入口になっている。
   - 医療従事者は「公的機関」「論文」では探さず、「今何に困っているか」で探すため、この設計は合理的。

2. **検索とカテゴリ絞り込みの併用**
   - 単なるカード一覧より圧倒的に実務向き。
   - サイト名を知らなくても、症状・用途・タグで探せる設計になっている。

3. **リンクに活用シーンが付いている**
   - ただのブックマーク集との差別化ポイント。
   - 利用者が「なぜこのサイトを見るべきか」を即理解できる。

4. **出典クレジットを控えめに明記している**
   - 過度に目立たず、信頼性と透明性を上げている。
   - 医療情報系では、情報の再構成元を曖昧にしない姿勢は重要。

### 17.2 設計上の弱点

| 項目 | 評価 | リスク | 備考 |
|------|------|--------|------|
| セクションアンカーの視認性 | **要改善** | Low | sticky header を使用しているが、アンカー遷移時の `scroll-margin-top` がない。見出しがヘッダーに隠れる可能性あり |
| 検索状態の共有性 | **不適合** | Low | 検索語やカテゴリ絞り込みがURLに反映されない。ブックマーク・共有・再訪に弱い |
| 情報更新日表示 | **不適合** | Medium | 医療系リンク集であるにもかかわらず、「いつ確認した情報か」が画面上でわかりにくい |
| リンク監査導線 | **不適合** | Medium | 外部リンク切れ・移転・制度変更に備えた保守導線がない |
| 利用者対象の明示 | **要改善** | Low | 薬剤師・医療従事者向けであることは分かるが、「患者向けの医療判断サイトではない」ことをより明確にしてもよい |

**改善提案：**

```css
section[id] {
  scroll-margin-top: 96px;
}
```

また、フッターまたはヒーロー下に以下のような更新管理表記を入れると運用品質が上がる。

```html
<p class="meta-note">最終リンク確認日：2026-04-20</p>
```

## 18. 将来の脆弱化ポイント（運用リスク）

このサイトは**現時点では安全**だが、将来の改修で一気に弱くなるポイントが明確にある。
ここを事前に言語化しておかないと、次の改修者が高確率で踏む。

| 項目 | 現在 | 将来リスク | 備考 |
|------|------|------------|------|
| データ管理方式 | ハードコード | High | JSON外部化・CMS化・スプレッドシート連携時にXSS/不正URL混入の入口になる |
| URLの信頼性 | 自前管理 | Medium | 将来、外部ソースを取り込むと `javascript:` や不正なプロトコル混入リスクが出る |
| 医療説明文 | 手動記述 | High | 将来、説明文が増えるほど薬機法・医療広告・誤認表現のリスクが上がる |
| 情報鮮度 | 作成時点依存 | High | 医療情報は時間経過で価値が落ちる。古いまま残ると信頼性が逆に下がる |
| CSS保守性 | 単一HTML内蔵 | Medium | スタイル量が増えると変更影響範囲が読みにくくなり、デグレしやすい |

### 18.1 具体的な追加対策

1. **URLバリデーションを入れる**
   - 将来データ外部化するなら、`http:` `https:` のみ許可する関数が必要。

```js
function safeUrl(url) {
  try {
    const u = new URL(url, location.origin);
    return ["http:", "https:"].includes(u.protocol) ? u.href : "#";
  } catch {
    return "#";
  }
}
```

2. **説明文の表現ルールを定義する**
   - 「推奨」「必須」「安全」などの断定語を乱用しない。
   - 「確認に使う」「参考にする」といった中立表現を維持する。

3. **リンク棚卸しの運用を決める**
   - 3か月〜6か月ごとにリンク切れ、移転、制度改定の棚卸しを行う。
   - 医療情報サイトでは、コードの安全性より更新運用の方が信用に直結する。

4. **HTML/CSS/JSの分離を見据える**
   - 今は単一HTMLで問題ないが、今後規模が増えるなら `style.css` / `data.js` / `app.js` に分離した方が安全。

## まとめ

このサイトは**静的HTMLとして非常に堅牢で軽量な設計**になっている。
セキュリティ上の深刻な問題はなく、主な課題は「公開メタ情報の整備（SEO・OGP）」「医療情報としての免責文」「ファビコン・アクセシビリティ細部」の3点に集約される。
High優先度の3項目は30分程度で対応できる。
